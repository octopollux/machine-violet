# Image Generation

The DM can render illustrated images ‚Äî a character reference sheet during setup, scene snapshots during gameplay, ad-hoc illustrations on player request. Rendering goes through whichever provider backs the `large` tier, gated by both provider capability and per-campaign consent.

**Setup renders are synchronous; gameplay renders are fire-and-forget.** During chargen the setup agent shows portrait drafts inline and the player iterates on them, so that path *awaits* the render. In gameplay a render takes 1‚Äì3 minutes at the backend's own (uncapped) quality, so blocking the turn on it would freeze play ‚Äî instead the DM's `generate_image` is **fire-and-forget**: it returns immediately and the finished image surfaces on a later turn, divorced from the one that requested it. See [Background render (gameplay)](#background-render-gameplay).

## Capability + consent

Image-gen is enabled for a turn iff:

- **Provider capability** ‚Äî `provider.getCapabilities(model).imageGeneration === true`. True for `openai-apikey` + `gpt-5` family (routes to `gpt-image-2` via the Images API) and for `openai-chatgpt` (routes to codex's built-in `image_gen` skill ‚Äî see [Provider backends](#provider-backends)). False for everything else (Anthropic, and the Chat Completions fallback used by custom OpenAI-compatible endpoints).
- **Campaign preference** ‚Äî `gameState.config.image_generation !== "off"`. The setup-agent asks the player a yes/no consent question right after world + system selection ([game-initialization.md](game-initialization.md)). The choice is stored in `config.json` and can be flipped per-campaign from the ESC menu's "Image generation" toggle.

When either is false, the `generate_image` tool is omitted from the DM's toolset and the model has no way to invoke it. There is no "graceful fallback" path on the model side ‚Äî the model simply doesn't see the tool.

## Tool dispatch

`generate_image` is a **function tool** (not a hosted/server-side tool). The model calls it like any other tool; the host runs `provider.generateImage()` and persists the bytes. How the rendered image reaches the player differs by call site (synchronous inline vs. fire-and-forget ‚Äî see below).

Two call sites:

| Site | File | Dispatch | Purpose |
|---|---|---|---|
| Setup-agent portrait loop | `packages/engine/src/agents/subagents/setup-conversation.ts` (`dispatchGenerateImage`) | **Synchronous** ‚Äî awaits the render, returns `_tui: display_image` so the draft shows inline immediately | Character reference-sheet drafts during chargen (a landscape multi-angle turnaround ‚Äî see [Visual style guidance](#visual-style-guidance)). Player iterates; `set_portrait` promotes a chosen draft. |
| DM gameplay | `packages/engine/src/agents/game-engine.ts` (`dispatchGenerateImage`) | **Fire-and-forget** ‚Äî kicks a background render, returns an immediate ack; the image surfaces on a later turn | Scene snapshots, character close-ups, player-requested illustrations. |

Both share `handleImageGenerated` in `packages/engine/src/agents/image-handler.ts` for on-disk persistence ‚Äî same naming, same sidecar JSON, same downstream consumers.

The legacy hosted-tool path (`image_generation_call` items in the Responses API) is explicitly **not** used. The provider has a tripwire (`image_gen:legacy_hosted_item_ignored`) that fires if such an item ever shows up ‚Äî see [the dispatch memory](../packages/engine/src/providers/openai.ts) for context.

## Provider backends

`provider.generateImage(req)` is the single seam the dispatchers call. Two providers implement it, with very different mechanics behind the same `GenerateImageRequest ‚Üí GenerateImageResult` contract:

| Provider | Mechanism | Auth / billing |
|---|---|---|
| `openai-apikey` | One `gpt-image-2` REST call (`openai.ts` `openaiGenerateImage`): `client.images.generate({ quality, size })` for text-to-image, or `client.images.edit({ image: [refs], quality, size })` when reference portraits are attached (image-to-image ‚Äî see [Reference conditioning](#reference-conditioning-image-to-image)). Empty renders retry up to 3√ó. | Platform API key + credits. |
| `openai-chatgpt` | Drives a **scoped codex turn** that calls codex's built-in `image_gen` skill, then harvests the `imageGeneration` item codex emits (`provider.ts` `generateImage`). | ChatGPT subscription ‚Äî **no API key**; billed to the plan's Codex allocation. |

The `openai-chatgpt` path is the happy path for most users (sign in with a ChatGPT subscription, no metered API spend). There is **no fallback** between providers: a ChatGPT-account DM never silently routes image-gen to a metered API-key provider. If the codex render fails, the dispatcher returns an `isError` tool_result the DM recovers from ‚Äî it does not escalate to a more expensive backend.

Why the codex path looks the way it does (all spike-confirmed on codex 0.133.0):

- **No direct RPC for image_gen.** It's a model-driven skill, not an app-server method. So `generateImage` issues its own `thread/start` + `turn/start`, separate from the DM's conversation thread, asking the model to render the prompt.
- **The bytes arrive two ways ‚Äî inline, and on disk.** Codex emits an `item/completed` with `type: "imageGeneration"` whose `result` field holds the raw base64 PNG, **and** writes the same PNG under `<codexHome>/generated_images/<sessionId>/`. The inline path is the zero-I/O fast path (`extractGeneratedImage` reads `result` + `revisedPrompt`), but that `item/completed` line is multi-MB of base64 on a *single* newline-delimited JSON record, and a render's worth of bytes (2‚Äì3 MB PNG ‚Üí ~3‚Äì4 MB base64) can be **dropped or corrupted on the stdio pipe** ‚Äî `JSON.parse` then throws and the line is discarded, so the turn "completes" having emitted nothing we caught, even though the GPU work succeeded and the file is on disk. The robust fix is to fall back to **reading the PNG off disk** from that per-session dir (`readNewestPngAsBase64`): the dir is keyed by the ephemeral render thread's unique `sessionId` (learned from `thread/start`; the dir base is `codexHome` from the `initialize` handshake), so the newest PNG in it is unambiguously this render's, with no race against concurrent renders. This is no longer a copy we ignore ‚Äî it's the canonical source when the wire drops the inline bytes. (Confirmed: a complete, valid 2.16 MB PNG landed on disk at the exact moment our inline harvest reported "no data.") Once the bytes are in hand, `renderImageOnce` deletes that per-session dir in its `finally` (`removeGeneratedImageDir`, best-effort/never-throws) ‚Äî we persist our own copy into the campaign, so codex's copy is pure redundancy that would otherwise accumulate ~2‚Äì3 MB per render forever.
- **Tightly scoped.** The turn runs `sandbox: "read-only"` with terse `developerInstructions` ("call image_gen once, no shell, no file ops, no prose"). Without this the model treats the request as a coding task and wastes a round trying to `Copy-Item` the file to satisfy "save it" phrasing (harmless under read-only, but slow).
- **Nested-turn safe.** `generateImage` runs *while the DM turn that called `generate_image` is paused* awaiting our tool reply. Codex processes the two threads independently over the JSON-RPC pipe, so the inner render turn completes and the outer turn resumes ‚Äî no global serialization / deadlock.
- **No quality knob ‚Äî `effort` is a no-op here; only `aspect` steers.** The built-in `image_gen` tool exposes **no** explicit size/quality param over the wire ‚Äî codex sends the hosted `image_generation` tool bare, so the backend renders at its own `auto` quality, and that's not reachable from the RPC, codex config, or the model's tool call (confirmed against codex 0.133‚Äì0.142; upstream [openai/codex#20839](https://github.com/openai/codex/issues/20839) and [#19175](https://github.com/openai/codex/issues/19175) are open with no fix). A live A/B (2026-06-22, 8 renders of one subject across fidelity/size phrasings) made the consequences concrete: every render came back at a **fixed ~1.57 MP** (`square` ‚Üí 1254√ó1254, `landscape` ‚Üí 1536√ó1024 ‚Äî same pixel count, just reshaped), an explicit "512√ó512 thumbnail" was echoed in the backend's `revisedPrompt` but ignored, and render time/plan-usage did **not** track low‚Üîhigh fidelity wording. So `buildImagePromptText` folds in **only `aspect`** (the one lever the backend honors ‚Äî orientation, not pixel count). `effort` is deliberately **not** steered into the prompt: it would change at most how *detailed* the image looks (a content/aesthetic effect, never cost), so it rides the request and echoes back via `effortUsed` for cross-provider contract parity only. `aspectUsed` echoes the requested value. To re-check whether a codex bump finally exposes the knob, dump the protocol with `codex app-server generate-json-schema` (look for any image quality/size field) and run a quick render A/B comparing output dimensions across `effort` levels.

## Abstract knobs

`packages/engine/src/providers/types.ts` defines two abstract enums the model picks from:

- **`ImageEffort`** = `"draft" | "standard" | "quality" | "showcase"`. **Per-provider reality differs:** on the **openai-apikey** path it's a real lever, mapping to `quality: "low" | "medium" | "high" | "high"` on the `images.generate` call; on the **ChatGPT/codex** path it's a **no-op** (no quality param exists ‚Äî see [Provider backends](#provider-backends)), accepted and echoed via `effortUsed` but not steered into the render. The knob stays in the abstract contract regardless: it's load-bearing on openai-apikey and ready for any future image-capable backend (e.g. if Anthropic ships image gen) without a schema change. The DM picks it per call; on a codex-backed campaign the choice simply has no render effect.
- **`ImageAspect`** = `"portrait" | "landscape" | "square"`. Use-case-named, not pixel-named ‚Äî `portrait` is character close-ups, `landscape` is scenes (and the chargen reference sheet, whose three side-by-side views need the width), `square` is objects. This is the **one render knob that works on every backend, including codex** ‚Äî though on codex it reshapes a fixed ~1.57 MP pixel budget rather than changing the pixel count.

The chargen reference sheet is always rendered at `effort: "standard"` and `aspect: "landscape"` ‚Äî pinned in the setup dispatch (`dispatchGenerateImage`), not chosen by the model, since chargen has exactly one canonical config: medium quality is the ceiling a character card needs and it's fast enough to iterate on, and landscape gives the three angles room. (The setup `generate_image` tool exposes only `prompt`; the DM's gameplay tool keeps the effort/aspect knobs.) The DM defaults to `quality` (high quality at a reasonable render time) for scene snapshots and player-requested illustrations, drops to `standard` for quick/minor inserts, and reserves `showcase` for rare set-piece hero shots (`packages/engine/src/prompts/dm-directives.md`).

## Reference conditioning (image-to-image)

The DM can ask a render to **match a player character's established look** by naming them in the `generate_image` tool's optional `reference_characters` array (a list of character names). The named characters' portraits are fed to the renderer as image-to-image references so the depicted character comes out on-model (face, build, outfit) instead of a fresh invention.

**Opt-in, never default.** A portrait reference biases the *whole* render toward that character, which is wrong for a wide scene they're barely in (or not in at all) ‚Äî so references are attached only when the DM explicitly lists names. The tool description steers the model to use it only when a specific PC is clearly the subject and their likeness matters, and to leave it off for crowds/scenery.

**Name ‚Üí portrait resolution** (`loadCharacterReferences` in `packages/engine/src/agents/dm-portraits.ts`, called from `GameEngine.dispatchGenerateImage`): each name maps through `slugify` to `<campaignRoot>/characters/<slug>-portrait.png` ‚Äî the *same* file `loadDmPortraitMessage` injects into the DM's context. Resolution is deliberately forgiving: names with no portrait on disk (an NPC, a PC whose player declined image gen, a typo) are skipped silently, deduped by path, and an all-miss list omits the field entirely so a plain text-to-image render happens. So a bad name degrades to "no reference," never an error.

**Transport.** Both gpt-image-2 backends condition on the resolved portraits, and both append the *same* reference directive ‚Äî shared via `buildReferenceDirective` (`packages/engine/src/providers/image-reference-directive.ts`) so they behave identically. On the **`openai-chatgpt` / codex** path, each portrait rides as an `image` input item *ahead of* the text prompt on the render turn, `buildImagePromptText` appends the directive naming the referenced characters ("‚Ä¶the established appearance of X ‚Äî match that character's facial features, build, and outfit to the reference, but follow this description for pose, facial expression, setting, and framing (the reference carries only one neutral expression ‚Äî take the emotion from the description)"), and codex's `image_gen` conditions the render on them. On the **`openai` API-key** path, the portraits are uploaded (via `toFile`) as `Uploadable`s to gpt-image-2's `images.edit` endpoint with the identical directive appended to the prompt (a no-reference render uses plain `images.generate`). **Facial expression is deliberately on the description side**, not the reference side: a saved portrait carries one neutral expression, so without this the depicted character would inherit that flat face regardless of the scene's emotion. The DM-facing tool description and `dm-directives.md` reinforce it from the other end ‚Äî the model is told to name each character's expression in the `prompt`. **Validated** (codex path): a reference render reproduced a PC's distinctive features (hair, freckles, hex-pattern pauldron, outfit) in a brand-new scene the prompt described from scratch, where the no-reference control was an unrelated character.

**Cost / caveats.** Image-to-image is materially slower than text-to-image (gpt-image-2 backend cost ‚Äî a reference render is minutes, vs. roughly a second for plain text-to-image). Earlier the renderer instruction told the model to "match each character's face to their reference," which sometimes made it spend a reasoning pass *analyzing* the attached portrait and then reply with text instead of calling `image_gen` at all (a wasted multi-minute attempt the bounded retry rescued). #596 reframed the instruction ‚Äî references are consumed by `image_gen` automatically, so the model is told *not* to view/analyze them, its only job is to call the tool once ‚Äî and a live 8-render reference batch afterward showed **0 text-replies (8/8 first-attempt images, avg 1.00 attempts)** while conditioning still held (the depicted character stayed on-model). The bounded retry (see [Provider backends](#provider-backends)) stays as the safety net for the residual intermittent case. Lowering the render turn's reasoning effort would be the other lever, but `"low"` is the **floor** ‚Äî the backend rejects `image_gen` at `reasoning.effort: "minimal"` (HTTP 400), so it isn't available. Net: references work and come out on-model, but each render is minutes ‚Äî reserve them for deliberate "show *this* character" moments. To re-measure the first-attempt rate after any renderer-instruction or effort change, run the probe: `node --import tsx/esm packages/test-harness/bin/ref-render-rate.ts` (issue #599).

## Background render (gameplay)

In gameplay `GameEngine.dispatchGenerateImage` is **fire-and-forget**, modeled on `update_portrait` (see [Mid-campaign portrait revision](#mid-campaign-portrait-revision-update_portrait)):

1. It validates inputs, snapshots the **current scene** (so the image files under the scene it was requested in even if the scene transitions before the render lands), kicks the render on a **tracked promise** (`inFlightImageRenders` + `trackImageRender`, `.catch` logs ‚Äî never a detached `void`), and returns an **immediate text ack**. The DM keeps narrating without waiting.
2. When the (1‚Äì3 minute) render completes, the bytes persist via `handleImageGenerated` and the finished image pushes onto `pendingImageDisplays`.
3. At the **next turn boundary**, `processInput` drains that queue (`consumePendingImageDisplays`): each image is broadcast to the client as a `display_image` TUI command (after the turn's narration completes, so it lands *below* the prose, not spliced mid-stream) and appended to the display log behind a separator divider, so on reload it reads as a standalone illustration of an earlier beat rather than part of that turn's narration.

So a gameplay image is **divorced from the turn that requested it** ‚Äî by design, so a slow render never blocks the player. Render-time errors (refusal, network) can't reach the DM anymore (the turn is long over), so they're logged and the image is silently dropped ‚Äî the DM never claimed it appeared, because the prompt forbids narrating the image as present.

**Resting activity indicator.** While a background render is in flight and the engine sits between turns, the engine state rests in `generating_image` (driven off `inFlightImageRenders` via `refreshRestingImageState`/`restingState`), so the client shows "The DM is creating an image‚Ä¶". This is a *resting* state ‚Äî `processInput` accepts it as a valid entry point, so input is **never** blocked while an image cooks. For an explicit `player_request` render this doubles as the "it's coming" acknowledgment.

**Teardown.** `awaitPendingImageRenders` (for tests/graceful teardown) awaits in-flight renders; `endSession` best-effort flushes any already-completed-but-unshown images to the display log (`flushPendingImageDisplaysToLog`) so they reappear in the next resume's scrollback. It does **not** await still-in-flight renders (they can take minutes; the bytes are on disk regardless).

**Pacing.** The DM self-paces via prompt guidance ‚Äî a campaign-scope target of ~N images per 100 player exchanges (`image_cadence_per_100`, default 12, interpolated into `dm-directives.md` as `{{imageCadence}}` by `buildDMPrefix`). Player-requested images render regardless and don't count against the target. The DM is also told to fire one establishing image with the PC(s) in frame in the campaign's first few exchanges. There is no longer any coupling between image generation and scene transitions.

The setup path still uses the `_tui` broadcast: the bridge dispatches a turn's tool calls concurrently via `Promise.all`, and tool results carrying a `_tui` field broadcast immediately. Extraction is gated on `tuiToolNames.has(toolName)` ‚Äî tools not in `TUI_TOOLS` (`packages/engine/src/agents/agent-loop.ts`) have their `_tui` silently ignored. `generate_image` stays in the set (the setup portrait loop returns `_tui: display_image`); a regression test in `agent-loop.test.ts` guards against its removal.

## Inline rendering (TUI)

The client draws `display_image` lines inline with an MV-owned renderer (`packages/client-ink/src/tui/image/`), which replaced `ink-picture`. Terminal graphics protocols paint pixels out-of-band at absolute screen positions with no clipping and no z-ordering against Ink's text grid, so the renderer owns composition with the live, scrolling, layered TUI.

**Protocol selection.** At startup `start-client.ts` probes the terminal (`detectGraphicsCapabilities` in `tui/image/capabilities.ts`) and picks the best of three real graphics protocols ‚Äî **kitty > iTerm2 > sixel** (`pickProtocol`). The detection path is layered: pure per-query parsers ‚Üí a pure assembler ‚Üí a declarative `TERMINAL_QUIRKS` table ‚Üí `pickProtocol`. Per-terminal workarounds are one table entry each (e.g. iTerm.app's partial kitty emulation is suppressed in favor of its native protocol). A terminal with **no** graphics protocol renders nothing inline ‚Äî the image still exists at full resolution in the HTML transcript export (see Disk + transcript).

**Drivers** (`tui/image/drivers/`) implement a small `ImageDriver` interface in two families:

| Protocol | Family | Mechanism |
|---|---|---|
| kitty | placement | transmit raw RGBA once by image id, display any band via a native source-rect crop; true-color, flicker-free re-place |
| iTerm2 | blit | per-band PNG (`OSC 1337`), sized in **cells** so it lands on the text grid; true-color |
| sixel | blit | per-band, one shared quantized palette (no requantize-on-scroll); **alpha flattened to opaque before quantizing** ‚Äî a transparent letterbox otherwise scatters the whole palette into noise |

**Sizing.** The parent (`NarrativeArea`) passes **bounds** ‚Äî the full content width and a height cap that's a fraction of the narrative viewport (`imageHeightCapFraction`: `0.9` for every intent) ‚Äî not a fixed footprint. An image is the focus of the moment and gets nearly the whole pane. (This cap used to be keyed off intent ‚Äî `character_portrait` got a tighter `0.6` because the setup portrait was a tall front-facing close-up ‚Äî but the setup portrait is now a *landscape* multi-angle reference sheet, wide-not-tall, so it shares the scene budget; the only other `character_portrait` render, the DM's silent `update_portrait`, is never displayed. The intent param is retained so a future tall-portrait intent could reintroduce a tighter cap.) `InlineImage` reads the image's true dimensions (sharp metadata, header-only) and `fitImage` (`geometry.ts`) reserves exactly the scaled footprint at the image's real aspect: landscapes fill width, portraits fill height up to the cap. Cell aspect (cells are ~2:1 tall:wide) is folded into the math so a square image reads square on screen. This replaced an earlier scheme where the parent reserved a 16:9 box and the component `fit:contain`-ed into it ‚Äî that letterboxed twice (parent box + component padding) and needed an `isPortrait`/`intent` hack to special-case tall portraits. There's one reflow when metadata lands (a wide default holds the slot until then, so non-portrait images don't move).

**Composition.** The image decodes + resizes once (sharp), then a painter registered with the sync-write combiner re-blits the visible band inside the same DEC-2026 synchronized-output frame Ink just drew ‚Äî flicker-free. The painter is the **single authority for both position and pixels**, and it runs at the only correct moment: the combiner splices it in just before ESU, which Ink emits from `onRender` *after* `calculateLayout()`. So the painter measures the slot's on-screen row fresh each frame and reads the row for the frame being drawn. (Measuring in a `useLayoutEffect` instead ‚Äî React's commit, *before* Ink lays out ‚Äî lags one frame: invisible on a 1-row scroll, but a multi-row PgDn jump paints the image a full page behind the text and strands it there once the scroll settles. So there's no layout-effect repositioning, and `NarrativeArea` no longer threads a changing prop to break the image line's `React.memo` ‚Äî the painter, which runs every Ink frame regardless, re-measures and re-clips itself, which also drops the per-tick re-render of off-screen image lines.) `visibleBand` clips the footprint to the narrative viewport, so the image never overflows the modeline above or the input line below ‚Äî this **partial (vertical) crop is for the viewport's top/bottom edges only**. The band re-encodes in-frame whenever its identity changes (cheap for all drivers ‚Äî sixel/iTerm2/kitty are all real-time) and reuses the cached payload when it hasn't, so a static image never re-encodes. The painter is also **idempotent across frames**: it compares a cheap signature (footprint + source band + raster generation) against the frame it last blitted and emits nothing when they match. Combined with Ink's **incremental rendering** (enabled in `start-client.ts` ‚Äî Ink rewrites only the lines whose text changed instead of erasing + redrawing the whole frame), an unrelated re-render ‚Äî the activity line's once-a-second elapsed counter, a tool glyph landing ‚Äî no longer clobbers the image's blank slot rows, so its (up to ~MB-sized sixel/iTerm2) payload isn't re-pushed to the terminal every tick. The skip is **gated on incremental rendering** (a flag `start-client` sets in lock-step with the render option): the standard full-redraw renderer clobbers the image every frame, so there the painter must re-blit unconditionally or the image would vanish.

Modal occlusion is **all-or-nothing**: the painter reads the live occluder spans (`OcclusionProvider` / `useRegisterOcclusion`, registered by `CenteredModal` + `OverlayPane`) and hides the whole image if any span overlaps its band, restoring it when the modal closes. The hide/show is **edge-triggered** via `useOcclusionChange`: the registry notifies subscribers on every register/unregister, and the image schedules a (deferred, coalesced) repaint. This is deliberate ‚Äî sixel blits are slow and Ink frames get dropped, so relying on the frame for that exact render to carry the occlusion change misses the edge (the image hid on the first modal open, then stayed stranded over the modal on later opens). The deferral lands the repaint after the modal registered its span (no flash); the coalescing caps it to one write per burst. The inline `ChoiceOverlay` registers no span, so it leaves the image visible. (Partial *modal* cropping ‚Äî showing the rows a modal leaves free ‚Äî was built and removed: interior splits and re-encode timing weren't worth the complexity for a transient overlay.) The painter's vacated-row erase is passed the occluder spans and **skips any row a modal owns** ‚Äî otherwise the erase, fired the instant a modal covers the image, would blank the modal where the image had been (the "modals hidden by image" bug).

The style governs how the **character** renders only: the engine appends a reference-sheet framing AFTER the style line (`PORTRAIT_SHEET_FRAMING`), so the scene-oriented catalog styles neither stage the turnaround in an environment nor drag along their chrome (captions, letterbox/frames, vignettes, HUD/timestamp overlays) ‚Äî the sheet stays a clean reference card on plain black. This keeps the character card consistent with the in-game art the DM later conditions on it (the seed's `image_style` also drives that, via `<!--include:Image.<style>-->` in `campaign_detail` ‚Äî see [format-spec ¬ß10.8](format-spec.md#108-visual-style-image_style)).

**The setup portrait is a multi-angle reference sheet, not a single front-facing pose.** A lone camera-facing portrait is a strong prior: routed into the DM's context as both a general reference and an image-to-image conditioning input (`loadCharacterReferences`), it biased *all* in-game renders toward the character standing and posing for the camera ‚Äî protagonists who look like they're holding still for a photo while fighting a dragon. So the chargen prompt now asks for a **landscape** sheet showing the same full-body character from three angles side by side ‚Äî a front view, a front-left three-quarter, and a rear-right three-quarter ‚Äî against a plain black background; the campaign's visual style (above) governs how the character is rendered, but the sheet stays a clean turnaround, not a staged scene. The extra angles give the conditioning path a fuller sense of the character (build, outfit, and silhouette from several sides) so generated scenes can place them in any pose. Because the sheet shows three angles at once, any **one-sided feature** has to stay put: `PORTRAIT_SHEET_FRAMING` tells the model to keep an asymmetric feature (a prosthetic or missing limb, a cybernetic eye, an eyepatch, a scar, a holstered weapon) on the *same* side of the body in every view, and the setup agent is instructed to **commit to a definite side** when the player's description leaves it ambiguous ("a robot arm" ‚Üí "a robotic LEFT arm") rather than let it drift to a different side per angle. This is why setup uses `aspect: "landscape"` (the standard scene aspect) rather than the old `portrait` framing; the canonical on-disk path is still `characters/<slug>-portrait.png` ‚Äî "portrait" names the role, not the framing.

PC portraits are injected into the DM's context as `image_input` ContentParts in the non-ephemeral cached prefix via `loadDmPortraitMessage` (`packages/engine/src/agents/dm-portraits.ts`) ‚Äî the model has a visual reference for each PC's appearance without any per-turn token cost beyond the initial cache write (~85 input tokens per image with `lowDetail`).

**The through-routed copy is downscaled, not the full-res original.** The canonical on-disk portrait is a full-resolution PNG (~1‚Äì3 MB) ‚Äî that fidelity is kept for the image-to-image reference path (`loadCharacterReferences`) and TUI display. But injecting it base64-inline means re-uploading megabytes across the provider boundary on every cache write, and it bloats the `.debug/context/dm.json` dump past readability. `loadDmPortraitMessage` runs each portrait through `downscalePortraitForContext` ‚Äî a sharp resize to ‚â§512px longest edge re-encoded as **WebP q90** ‚Äî before base64-encoding. 512px matches what `lowDetail` already caps the model's perception at, so the model sees identical input at a fraction of the bytes (~1.8 MB ‚Üí tens of KB). Computed once per engine lifetime (the prefix message is cached). On any sharp failure it falls back to the original bytes as PNG rather than dropping the portrait.

All three DM-tier providers transmit these `image_input` parts to the model: `openai.ts` (`toResponsesInput`) emits Responses-API `input_image` items; `openai-chatgpt`'s `messageToResponsesItems` does the same for the codex `thread/inject_items` replay; and `anthropic.ts` (`toAnthropicMessage`) emits a base64 `image` block. One codex-specific gotcha: its Responses deserializer accepts only `detail: "high" | "original"` (it rejects the OpenAI SDK's `"low"`/`"auto"` outright), so `messageToResponsesItems` maps `lowDetail ‚Üí "high"` (codex's downscaled floor) and full-detail ‚Üí `"original"`. Anthropic has no per-image detail tier ‚Äî it auto-downsamples and bills by pixel area, which is the other reason the through-routed copy ships pre-shrunk. Whether the DM actually *uses* the portrait in a generated scene is left entirely to the model ‚Äî no prompt directive pushes the PC into images.

### Mid-campaign portrait revision (`update_portrait`)

A static setup portrait drifts stale as the story changes a PC's look (a lost boot, a new scar, a traded coat) ‚Äî it drags the DM's concept of the character and makes future scene renders off-model. The `update_portrait` tool (registered alongside `generate_image` when image-gen is on) lets the DM **silently** revise a PC's portrait. It is *not* a scene render and shows the player nothing.

`dispatchUpdatePortrait` (`game-engine.ts`) re-renders the portrait via `provider.generateImage`, image-to-image **conditioned on the character's current portrait** (so identity carries forward and only the change applies). It re-renders in `aspect: "landscape"`, and `buildPortraitRevisionPrompt` (`dm-portraits.ts`) explicitly asks for the **same three-view turnaround layout** ‚Äî otherwise the first revision would collapse the multi-angle reference sheet to a single pose and lose the angles in-game art conditions on. Three deliberate properties:

- **Versioned, not overwritten.** `commitPortraitRevision` (`dm-portraits.ts`) archives the outgoing portrait to `characters/portrait-history/<slug>-NNN.png`, then overwrites `characterPortrait` ‚Äî a stable pointer that always holds the current look. Old portraits are never destroyed; only the current pointer is read back into context.
- **Background, never blocking.** The tool returns an immediate text ack (which persists the DM's intent in conversation) and runs the render on a tracked promise (`inFlightPortraitRenders`, `.catch` logs ‚Äî never a detached `void`). The DM narrates the change immediately and doesn't wait.
- **The rendered image is returned to the DM "for free."** On completion the new portrait (downscaled via `downscalePortraitForContext`) is stashed in `pendingPortraitInjections`; the **next real player turn** folds it into that turn's user message as an `image_input` part (the OOC-summary pattern, `consumePendingPortraitInjections`) ‚Äî so the DM sees its actual work and it persists in history, with no dedicated turn or API call. It rides as a *user message*, not the tool_result, because OpenAI/codex `function_call_output` is string-only and can't carry an image.

The stale prefix portrait (`messages[0]`) is left untouched mid-scene ‚Äî no cache bust ‚Äî because `transitionScene` clears the conversation anyway (`stepPruneContext`), so it resets `cachedPortraitMessage` there to reload the current portrait into the new scene's prefix and drops any pending injection. The `generate_image` reference path (`loadCharacterReferences`) reads the pointer fresh per call, so new scene renders pick up the revised look immediately.

## Disk + transcript

All generated bytes land in `<campaignRoot>/campaign/images/` (or `<setupRoot>/campaign/images/` during chargen). The basename is keyed off the image's intent (see `pickBasename` in `image-handler.ts`):

| Intent | Basename pattern |
|---|---|
| `scene_snapshot` | `scene-NNN-<scene-slug>-<timestamp>.png` |
| `player_request` | `request-<timestamp>.png` |
| `character_portrait` | `portrait-draft-<timestamp>.png` (setup-agent only; on `set_portrait` the chosen draft is copied to `<setupRoot>/characters/<character-slug>-portrait.png` ‚Äî inside the `__setup__` scratch campaign, not the real one) |

Each image gets a `.json` sidecar alongside it with `id`, `intent`, `timestamp`, `mimeType`, optional `sceneNumber` / `sceneSlug`, and the OpenAI `revisedPrompt` if the provider returned one. The sidecar is never load-bearing for rendering ‚Äî it exists purely as an audit record for debugging.

### Portrait handoff at campaign creation

`set_portrait` does **not** write into the live campaign. During chargen the setup agent runs against a `__setup__` scratch campaign, so `dispatchSetPortrait` (`setup-conversation.ts`) copies the chosen draft from `<setupRoot>/campaign/images/<draft_filename>` to `campaignPaths(setupRoot).characterPortrait(name)` ‚Äî i.e. `<setupRoot>/characters/<character-slug>-portrait.png`, still inside the scratch tree. (It sanitizes `draft_filename` to a bare basename and `mkdir`s `characters/` first, since the scratch tree starts minimal.)

The transfer into the real campaign happens later, inside `buildCampaignWorld` (step 9, `packages/engine/src/agents/world-builder.ts`). After scaffolding the new campaign directory, it reads `<campaignsDir>/__setup__/characters/<character-slug>-portrait.png` and, if that file exists, writes the bytes to the new campaign's `characterPortrait` path (`<campaignRoot>/characters/<character-slug>-portrait.png`). If the source is absent ‚Äî the no-portrait case, where the player declined image generation or `set_portrait` was never called ‚Äî the copy is silently skipped and the campaign starts without a portrait; the failure path inside the copy is likewise non-fatal. This copy is the bridge between the setup-agent's chargen path and the gameplay path: `loadDmPortraitMessage` in `dm-portraits.ts` reads the very same canonical `characterPortrait` location to inject the PC's likeness into the DM's context, so when the source file is missing it simply finds nothing to inject.

HTML transcript export (`packages/client-ink/src/commands/transcript.ts` ‚Äî `loadImageBytes` + `buildTranscriptHtml`) reads each referenced PNG and embeds it as a `data:image/png;base64,...` URI inline in the output HTML. The exported file is fully self-contained ‚Äî no sidecar PNGs, no external links, full on-disk quality. Missing files (moved/deleted between gameplay and export) degrade gracefully to `[image unavailable]` placeholders.

Once `buildTranscriptHtml` has produced the HTML string, the client persists it via `PUT /session/transcript` (`saveTranscript` in `packages/client-ink/src/api-client.ts`). The route ‚Äî implemented in `packages/engine/src/server/routes/data.ts`, mounted under the `/session` prefix ‚Äî writes the body to `<campaignRoot>/game-transcript.html` through the engine's `FileIO`, same as every other campaign write. It carries a per-route `bodyLimit` of **50 MB**, explicitly overriding the server-wide 1 MB default: long campaigns (around the 60-turn mark) produce transcript HTML that exceeds 1 MB and would otherwise be rejected with HTTP **413 Payload Too Large**, silently failing to save. The override is scoped to this single route so every other endpoint keeps the tighter default.

Inlined images are clickable: a small vanilla-JS shadowbox (no dependencies) fills the viewport with the image on a black backdrop. Esc or a click on the backdrop closes it; clicks and right-clicks *on the image itself* are left to the browser so the reader can save it / open it in a new tab. The overlay scaffold (`#shadowbox` + handler) is always emitted; image elements get a `zoomable` class. The images are also keyboard/screen-reader accessible ‚Äî `tabindex="0"` + `role="button"` + an `aria-label`, with Enter/Space opening the shadowbox when one is focused.
